#### The Sub-Agent System: Hierarchical and Constrained

Found comprehensive documentation on OpenClaw's sub-agent capabilities. The architecture is elegantly constrained:

**Two modes of operation:**
1. **One-shot runs** (`mode: "run"`) - Spawn a sub-agent, it completes its task, announces back to parent, then terminates
2. **Persistent sessions** (`mode: "session"` with `thread: true`) - Bind to Discord threads for ongoing conversation

**Three depth levels (when `maxSpawnDepth: 2`):**
- **Depth 0**: Main agent - can spawn sub-agents
- **Depth 1**: Orchestrator sub-agent - can manage children, receives coordination tools (`sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`)
- **Depth 2**: Leaf worker - no session tools, cannot spawn further children

This hierarchical structure prevents runaway delegation chains while enabling the coordinator-specialist pattern.

#### The Coordinator-Specialist Pattern

The most reliable multi-agent pattern for OpenClaw:

```
Coordinator (owns MEMORY.md)
    ↓ delegates via sessions_send
Specialist A (stateless, no memory) → returns result
Specialist B (stateless, no memory) → returns result
    ↑ aggregates
Coordinator writes summary to memory
```

Key constraints that make this work:
- **Coordinators cannot recurse** - If a task comes back from a specialist, aggregate and close, don't re-delegate
- **Specialists are stateless** - No persistent memory across tasks to prevent context contamination
- **Tool restrictions** - Specialists can be denied `sessions_send` entirely to prevent delegation loops
- **Max children per agent** - Default 5 active children prevents fan-out explosions

#### File-Based Shared State

For simple coordination without complex infrastructure, the pattern uses shared workspace files:
- `goal.md` - Current task and decomposition
- `plan.md` - Execution plan with subtasks and assignments
- `status.md` - State of each subtask (pending/in-progress/complete/blocked)
- `log.md` - Append-only execution log for auditing

This is refreshingly pragmatic. Instead of a database or message queue, you have files anyone can inspect to understand system state.

#### Preventing Failure Modes

The documentation was refreshingly honest about how multi-agent systems break:

**Infinite delegation loops**: Coordinator → Specialist A → Coordinator → Specialist A... 
- Fix: No-recursion rules in prompts, max depth limits, tool deny-lists

**Deadlocks on shared state**: Agent A waits for B, B waits for A
- Fix: Unique task IDs, status checks before claiming, per-run timeouts

**Cost runaway**: Each handoff adds token overhead; verbose summaries multiply costs
- Fix: Stateless specialists return structured output, coordinator handles final synthesis, use cheaper models for coordination

**Race conditions on files**: Two agents write to status.md simultaneously
- Fix: Append-only logs, idempotent writes, upcoming file locking support

#### Token Cost Reality Check

The documentation was blunt about costs:
> "Fewer than 10% of teams successfully scale beyond a single-agent deployment"

Main reasons: coordination complexity, uncontrolled token costs, poor state management.

Mitigation strategies that stood out:
- Keep specialists genuinely stateless (biggest cost lever)
- Summarize before storing in memory
- Use cheaper models for coordinator tasks (classification doesn't need top-tier intelligence)
- Set deliberate concurrency limits - parallel is faster but multiplies cost
